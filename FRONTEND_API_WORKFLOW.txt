LIMOSUD CARS - FRONTEND API WORKFLOW
====================================

Purpose
-------
This file explains how a frontend app should consume the implemented Laravel API without opening the generated Scribe docs.

Scope
-----
This document covers every endpoint currently defined in backend/routes/api.php for:
- Health
- Public API
- Admin authentication
- Admin dashboard
- Vehicles
- Customers
- Locations
- Reservations
- Payments
- Contracts
- Maintenance
- Expenses
- Alerts

Important implementation notes:
- Base API URL for local development: http://127.0.0.1:8000/api
- Generated API docs URL: http://127.0.0.1:8000/docs
- Admin auth type: Laravel Sanctum Bearer Token
- Public routes do not require authentication.
- Admin routes require Authorization: Bearer {token}, except POST /admin/auth/login.
- Public vehicle list/detail endpoints are available.
- Public and admin lookup endpoints are available.
- There are currently no user/role/permission CRUD routes exposed in routes/api.php.


1. PROJECT API OVERVIEW
=======================

Base URL
--------
Use this as the frontend API base URL:

http://127.0.0.1:8000/api

Docs URL
--------
http://127.0.0.1:8000/docs

Required JSON headers
---------------------
For JSON requests:

Accept: application/json
Content-Type: application/json

For protected admin requests:

Authorization: Bearer {SANCTUM_TOKEN}

For file upload requests:

Accept: application/json
Authorization: Bearer {SANCTUM_TOKEN}
Content-Type: multipart/form-data

Do not manually set the multipart boundary in browser apps. Let FormData/fetch/axios set it.


2. STANDARD RESPONSE FORMAT
===========================

Successful collection response
------------------------------
Paginated admin list endpoints usually return:

{
  "data": [
    {
      "id": 1
    }
  ],
  "links": {
    "first": "http://127.0.0.1:8000/api/admin/vehicles?page=1",
    "last": "http://127.0.0.1:8000/api/admin/vehicles?page=1",
    "prev": null,
    "next": null
  },
  "meta": {
    "current_page": 1,
    "from": 1,
    "last_page": 1,
    "path": "http://127.0.0.1:8000/api/admin/vehicles",
    "per_page": 15,
    "to": 1,
    "total": 1
  }
}

Some non-paginated collection endpoints return only:

{
  "data": [
    {
      "id": 1
    }
  ]
}

Successful single resource response
-----------------------------------
Most resource endpoints return:

{
  "data": {
    "id": 1
  }
}

Successful action response
--------------------------
Some actions return a plain JSON object:

{
  "available": true
}

Delete response
---------------
Soft delete endpoints return:

HTTP 204 No Content

Validation error - 422
----------------------
Laravel validation errors return:

{
  "message": "The given data was invalid.",
  "errors": {
    "email": [
      "The email field is required."
    ]
  }
}

Unauthorized - 401
------------------
Missing, expired, invalid, or revoked token:

{
  "message": "Unauthenticated."
}

Forbidden - 403
---------------
Authenticated user does not have the required permission:

{
  "message": "Forbidden."
}

Not found - 404
---------------
Invalid route model ID or missing resource:

{
  "message": "No query results for model [App\\Models\\Vehicle] 999"
}

Server error - 500
------------------
Unexpected backend error:

{
  "message": "Server Error"
}

Frontend error handling rules
-----------------------------
- 401: clear token and redirect to login.
- 403: show a permission error page or message.
- 404: show "not found" UI.
- 422: display field-level validation errors.
- 500: show a generic retry/contact support message.


3. FRONTEND WORKFLOW ORDER
==========================

Admin dashboard flow
--------------------
1. POST /admin/auth/login.
2. Store the returned access_token.
3. Call GET /admin/auth/me to fetch user, roles, and permissions.
4. Load GET /admin/dashboard/statistics.
5. Load GET /admin/dashboard/revenue and GET /admin/dashboard/expenses when charts need them.
6. Load lookup data if needed.
7. Load vehicles.
8. Load reservations.
9. Manage payments, contracts, maintenance, expenses, and alerts according to permissions returned by /admin/auth/me.

Public website flow
-------------------
1. Load GET /public/locations.
2. Load public lookups with GET /public/lookups.
3. Load public vehicles with GET /public/vehicles.
4. Load vehicle detail with GET /public/vehicles/{slug}.
5. Check availability with GET /public/vehicles/{vehicle}/availability or POST /public/reservations/check-availability.
6. Submit reservation with POST /public/reservations.


4. LOOKUP USAGE
===============

The backend uses lookup slugs for most form fields instead of hardcoded lookup IDs.

Use slugs when a request requires:
- status_slug
- transmission_type_slug
- fuel_type_slug
- document_type_slug
- location_type_slug
- payment_method_slug
- payment_type_slug
- payment_status_slug
- maintenance_type_slug
- expense_category_slug
- alert_type_slug
- alert_status_slug
- contract_status_slug where applicable

Current lookup endpoint status
------------------------------
Lookup endpoints now exist:
- GET /api/public/lookups
- GET /api/admin/lookups

Use public lookups for public website forms. Use admin lookups after login for admin forms.

Known seeded lookup slugs used by tests and backend logic include:
- vehicle_statuses: available, reserved, rented, maintenance, repair, out_of_service, sold
- transmission_types: manual, automatic
- fuel_types: diesel, gasoline, hybrid, electric
- reservation_statuses: pending, confirmed, in_progress, completed, cancelled, rejected
- payment_statuses: unpaid, partial_paid, paid, cancelled, failed, refunded
- payment_methods: cash, bank_transfer, credit_card, debit_card, check, online
- payment_types: reservation_deposit, rental_payment, remaining_balance, security_deposit, refund
- reservation_sources: website, phone, whatsapp, walk_in, partner, admin_manual
- location_types: agency, airport, hotel, custom
- maintenance_types: oil_change, tires, brakes, inspection, repair, cleaning, other
- expense_categories: maintenance, fuel, insurance, cleaning, parking, tolls, taxes, other
- alert_types: maintenance_due, document_expiry, insurance_expiry, payment_due, reservation_follow_up, vehicle_status, other
- alert_statuses: pending, seen, done, ignored
- document_types: passport, driving_license, cin, insurance, registration, technical_inspection, other
- contract_statuses: generated, signed, cancelled


5. FILE UPLOAD RULES
====================

Use multipart/form-data for:
- POST /admin/customers/{customer}/documents
- POST /admin/expenses
- PUT/PATCH /admin/expenses/{expense} if invoice is uploaded
- POST /admin/contracts/{contract}/signed

File constraints:
- Customer documents: field file, mimes pdf,jpg,jpeg,png,webp, max 10 MB.
- Expense invoice: field invoice, mimes pdf,jpg,jpeg,png,webp, max 10 MB.
- Signed contract: field signed_pdf, mime pdf, max 10 MB, optional.

Private paths:
- Do not display private file paths directly.
- Contract responses expose has_pdf and has_signed_pdf booleans.
- Use GET /admin/contracts/{contract}/download for generated contract downloads.
- CustomerDocumentResource currently returns file_path. Treat it as an internal path, not a public display URL, unless a public storage URL strategy is added later.


6. DATA TYPES AND UI NOTES
==========================

- IDs are numeric.
- Slugs are strings.
- Money values are decimal strings or numbers in JSON. Display as currency in the frontend.
- Dates accepted by validation are Laravel date-compatible strings.
- Recommended date format for date fields: YYYY-MM-DD.
- Recommended datetime format: YYYY-MM-DD HH:mm:ss or ISO 8601.
- Booleans can be true/false.
- Collection endpoints that paginate use ?page=1.
- Delete endpoints soft-delete records unless the module notes say otherwise.
- Payments have no DELETE route. Use POST /admin/payments/{payment}/cancel.

==========================

JSON fetch example
------------------
fetch("http://127.0.0.1:8000/api/admin/vehicles", {
  method: "GET",
  headers: {
    "Accept": "application/json",
    "Content-Type": "application/json",
    "Authorization": "Bearer " + token
  }
});

Multipart fetch example
-----------------------
const form = new FormData();
form.append("document_type_slug", "passport");
form.append("file", fileInput.files[0]);

fetch("http://127.0.0.1:8000/api/admin/customers/1/documents", {
  method: "POST",
  headers: {
    "Accept": "application/json",
    "Authorization": "Bearer " + token
  },
  body: form
});

- vehicle_id, pickup_location_id, dropoff_location_id

{

- 422: validation errors or overlapping active reservation.
- 404: invalid vehicle/location ID.
- 500: see Section 2.
Frontend notes:
- Public reservations are created as pending.
- Pending reservations do not change vehicle status.
- Show the reservation number after success.
